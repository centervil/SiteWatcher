# LLM開発支援ガイド（Cline連携版）

このドキュメントは、LLM（Clineなど）と連携して効率的に開発を進めるためのガイドラインです。

---

## 1. 基本的な開発サイクル

### 1.1 前日の活動履歴共有
- **目的**: LLMに現在のプロジェクト状況と進捗を把握させる
- **方法**: 箇条書きで日付と活動内容を簡潔に記載
- **例**:
```
昨日までの活動履歴は以下の通りです。
* 2025-02-24: タイムスタンプファイルの扱いについて改善
* 2025-02-25: Frontend Dockerfile Volumes 設定とホットリロード
* 2025-02-26: Frontend タイムスタンプ表示位置修正、JST対応、および表示問題解決
```

### 1.2 今日のテーマ設定
- **目的**: 開発の方向性を明確にし、対話をスムーズに進める
- **方法**: 具体的なテーマや目標を一文で明示
- **例**: `今日の開発テーマは、Zennとの連携機能の追加です。`

### 1.3 対話による開発進行
- **目的**: テーマに沿った開発を進める
- **方法**: 指示や質問を通じて開発を進め、議論と決定事項を記録
- **重要**: ユーザーからの指示がなくても、対話ごとに議論の流れと決定事項を開発日記に記録する
- **例**:
```
- Zenn CLIをインストールしてください。
- GitHub Actionsの設定ファイルを修正して、Zennへの公開処理を追加してください。
```

### 1.4 成果のバージョン管理
- **目的**: 開発成果を保存し、バージョン管理する
- **方法**: 変更をGitリポジトリにプッシュ
- **例**: `コードをリポジトリにプッシュしました。`

### 1.5 開発日記のまとめ
- **目的**: 日々の進捗を記録し、振り返りや共有に活用
- **方法**: 作業内容、学び、問題点などを構造化して記録
- **例**: `今日の開発日記を以下にまとめました。（開発日記の内容）`

---

## 2. 開発日記作成プロセス（2段階方式）

### 2.1 会話ログの記録 - 第1段階
- **保存場所**: `Documents/ProjectLogs/`
- **記録内容**: 
  - 指示内容、質問、試行錯誤の過程を忠実に記録
  - LLM自身の都合で行ったこと（ツール実行など）は含めない
- **フォーマット例**:
```markdown
## YYYY-MM-DD [テーマ名]

### 今日の開発テーマ
[テーマの説明]

### 開発プロセス
ユーザー: [質問/指示]
LLM: [回答/対応]
ユーザー: [続きの質問/指示]
...

### 課題と解決策
[課題、解決策、試行錯誤の記録]
```

### 2.2 Zenn公開用日記作成 - 第2段階
- **保存場所**: `articles/`
- **作成方法**:
  - 会話ログを元に、Zenn向けに再構成
  - 必要に応じて情報を補完
  - LLMに質問しながら完成させる
- **構造**: 後述の「開発日記の文章構成ガイドライン」に従う

### 2.3 プレビューと修正
- Zenn CLIでプレビュー表示
- 必要に応じて修正

### 2.4 公開
- 開発日記の公開設定を変更（`published: true`）
- Zennに公開

## 3. 開発日記の品質向上ポイント

### 3.1 構造と形式
- **frontmatter追加**: Zenn公開を考慮し、記事形式を統一
- **Zenn向けタイトル**: 可読性が高く、興味を引くタイトルに修正

### 3.2 内容と表現
- **所感欄の拡充**: 開発中の個人的な視点や感想を追加
- **親しみやすい文体**: 報告書調から親しみやすい文体へ変更
- **ユーザー視点での記述**: LLMではなくユーザーの視点で記述
- **実体験中心**: ユーザーの実際の作業や感想を中心に記述
- **具体的な作業と学び**: ユーザーの言葉で具体的に表現

---

## 4. 開発日記の文章構成ガイドライン

### 4.1 タイトル
- 内容を簡潔かつ魅力的に表現
- プラットフォームでの読みやすさを考慮
- **例**: "DockerfileとGitHub Pagesのタイムスタンプ問題 - build scriptへの集約"

### 4.2 Frontmatter
```yaml
---
title: "YYYY-MM-DD [テーマ名]"
emoji: "絵文字"
type: "tech"
topics: ["トピック1", "トピック2", "トピック3"]
published: false
---
```

### 4.3 導入（まえがき）
- テーマや背景、解決したい課題を簡潔に説明
- 読者の興味を引くプロローグ

### 4.4 検討事項（議論）
- 課題解決のための検討内容、初期案、代替案
- 思考プロセスの追跡

### 4.5 実行内容（実装）
- 実際の作業内容、コード修正、コマンド実行など
- コードブロックやコマンド例を積極的に使用

### 4.6 所感（考察）
- 開発中の感想、学び、苦労点、喜びなど
- 技術的内容だけでなく、感情や経験も共有

### 4.7 今後の課題（展望）
- 未解決の課題や今後の取り組み
- 明確なネクストアクション

### 4.8 結論（まとめ）
- 内容の簡潔なまとめ
- 主要な成果や達成の強調

## 5. LLMへの効果的な指示例

### 5.1 基本的な指示
- `〇〇ファイルを作成してください。`
- `〇〇ファイルに〇〇のコードを書いてください。`
- `〇〇ファイルの内容を〇〇に変更してください。`
- `〇〇コマンドを実行してください。`
- `〇〇ファイルの内容を読んでください。`
- `〇〇について質問があります。`

### 5.2 Zenn関連の指示
```
zenn new:article コマンドを実行して、新しい開発日記のテンプレートファイルを作成してください。
```

### 5.3 Frontmatter設定
- **title**: 開発日記のタイトル（例: "2025-02-28 Zenn自動公開の準備"）
- **emoji**: テーマに合わせた絵文字（例: "📝", "🚀", "🧪"）
- **type**: "tech"（技術記事）
- **topics**: 関連トピック（例: ["Zenn", "開発日記", "今日のテーマ"]）
- **published**: `false`（公開時に`true`に変更）

## 6. LLM活用のベストプラクティス

### 6.1 効果的なコミュニケーション
- **具体的な指示**: 明確かつ具体的に伝える
- **目的の共有**: 開発の目的や背景を伝える
- **質問の活用**: 不明点は積極的に質問する
- **反復的な対話**: 対話を重ねて深い洞察を得る

### 6.2 開発効率の最大化
- 一度に複数の指示を出すのではなく、段階的に進める
- 複雑な問題は分解して取り組む
- 定期的に進捗を確認し、方向性を調整する
- 成功パターンを記録し、再利用する

このガイドラインを活用し、LLMとの効果的な連携による生産性の高い開発を実現してください。
